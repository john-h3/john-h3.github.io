网站的分流

架构

最近发现网站并没有被百度给收录。我已经在页面上添加了谷歌和百度的相关代码，而且在谷歌上已经生效，显然不是代码的问题。经过在百度的资源管理中查看，发现的确是已经上传了链接，并手动添加了 sitemap 。

因为 coding.page 的静态页面总是莫名崩溃，我曾经将其迁移到 gitee 上去，这一次迁移后，我发现 gitee 的静态站点的自定义域名需要收费，于是就取消了部署。

结果之后意外能在百度上搜索到 gitee 站点的收录记录。显然是因为百度对于国外服务器的 ip 限制，导致即使上传了站点的链接，只要站点不是国内的站点那就不收录。而 coding.page 虽然是国内的，但是他们的静态页面部署使用的却是国外的服务器。现在，要想让我的网站被百度收录，就必须要依靠国内的服务器，但是 gitee 的自定义域名需要收费，现在决定将为百度的爬虫单独设置一个国内的服务器来收录。

访问流量的地域分流

对于来自不同地区的访问，由于国内外网络环境的差别需要提供不同的服务器。现在的计划是对于国内的页面访问将会到 Coding Pages，国外的访问会到 Github Pages，而来自百度爬虫的访问将会到单独提供的服务器。我们可以通过在域名解析中设置不同的解析线路来将访问流量分流。

单独服务器的架设

如果你有一台云服务器，那么最简单的方法就是直接将项目 clone 到服务器上部署，然后通过 nginx 提供服务即可。

负载均衡的示例配置：

upstream blog {
    server  localhost:port1  weight=1;
    server  localhost:port2  weight=1;
}

server {
    listen  80;
    server_name domain_name;

    rewrite ^(.*)$ https://$host$1 permanent;
}

server {
    listen 443 ssl;
    server_name domain_name;
    ...

    location / {
        proxy_pass  http://blog;
    }
}

server {
    listen port1;

    ...
}

server {
    listen port2;

    ...
}

至此，整个网站的分流修改就结束了。
